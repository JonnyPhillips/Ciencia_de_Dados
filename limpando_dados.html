Dividindo e Unificando Variveis

Dividir Variveis (separate)

Ns recebemos um banco de dados que combina dois pedaos de informao em uma varivel nico. Por exemplo, juntando ms e ano num formato combinado, ex. jan2013 Para filtrar, manipular ou calcular baseado no ms ou ano individualmente, sempre melhor organizar os dados em variveis separadas para cada pedao de informao. Isto um princpio de anlise dados: cada varivel merece a sua prpria coluna.

Para dividir uma varivel (coluna) em duas, usamos o verbo separate(). Os argumentos da funo so: (i) o nome da coluna que queremos dividir, (ii) os nomes das duas colunas novas, e (iii) como/onde queremos separar a coluna em duas. Pode verificar estes argumentos usando o help, ?separate. O (i) fcil. No (ii) temos que especificar dois nomes, um para cada nova coluna, ento no esquea de usar o c(), ex. c("mes","ano") para indicar que voc queria um vetor com mais de um elemento.

O (iii) o mais poderoso e complicado: Existem dois jeitos de separar uma varivel em duas - por posio, ou por caractere. Vamos ver por posio primeiro, que indica que a varivel tem um padro fixo definido pelo nmero de caracteres entre os dois componentes. Ex. jan2013 pode sempre ser separado depois do terceiro caractere, ento usamos um separador de 3.

Vamos mostrar com uma tabela pequena como uma varivel de ms-ano pode ser dividido por posio:

ex_separate <- tibble(ID=1:3,
                      Mes_Ano=c("jan2013","mar2009","out2015"))

ex_separate %>% separate(Mes_Ano, c("Mes","Ano"), 3)

Funciona, sim? Observe que o separate tira a varivel original do tibble; se quiser manter, tem que adicionar o argumento remove=FALSE dentro do separate.

s vezes o fornecedor de dados no to generoso com a padronizao das variveis. Como, por exemplo, dividimos os dados abaixo?

ex_separate_2 <- tibble(ID=1:3,
                      Mes_Ano=c("janeiro_2013","marco_2009","outobro_2015"))

No podemos usar posio dado que o tamanho do ms varia. Mas podemos pedir para o R buscar dentro do string de texto para o caractere especfico que divide o ms e o ano: _. O separate aceita este caractere em vez de nmero de posio:

ex_separate_2 %>% separate(Mes_Ano, c("Mes","Ano"), "_")

Muito poderoso! Veremos em uma aula futura como trabalhar mais com estes strings de texto complicados.

Unificar Variveis (unite)

Podemos fazer o inverso e unificar duas variveis para uma? Sim, fcil, mas temos que justificar o porqu. Normalmente faz mais sentido deixar os pedaos de informao distintos em variveis separadas. Pode ser que por motivos de apresentao queremos unificar duas variveis em uma. Por exemplo, para juntar nome e sobrenome em uma coluna para um relatrio. Veja abaixo o uso de unite() para devolver as nossas colunas de ms e ano para mes_ano. Os argumentos so basicamente os mesmos que separate(), sem o terceiro elemento do separador:

ex_separate_3 <- ex_separate_2 %>% separate(Mes_Ano, c("Mes","Ano"), "_")

ex_separate_3 %>% unite("Mes_Ano", c(Mes,Ano))

Observe que unite() insere um _ entre os componentes por padro; podemos especificar o nosso caractere preferido (ou nenhum) com mais um argumento na funo, por exemplo sep=", ".

ex_separate_3 %>% unite("Mes_Ano", c(Mes,Ano), sep=", ")

Habilidade Bsica de Programao: Aspas ou sem Aspas?

Voc j observou que estamos nos referindo s variveis em duas formas diferentes? s vezes com aspas "Varivel", e s vezes sem aspas Varivel. Como sabemos qual a usar? um tpico um pouco chato, mas importante.

Usamos sem aspas quando referimos a um objeto que j existe em nosso Environment, ou uma coluna j existente num tibble. Usamos com aspas quando referimos a um nome novo que estamos gerando naquele momento.

Ento em separate so as duas novas colunas que exigem de aspas, enquanto no unite a coluna nica e nova que exige aspas.

Ao trabalhar com caracteres especficos dentro de uma varivel/tabela, sempre temos que usar aspas.

O R meio tolerante e tenta ajudar em alguns casos se usamos o formato errado. Mas importante evitar erros e confuses se possvel.

Recodificao de Variveis

Recodificao de variveis uma habilidade fundamental na preparao de um banco de dados. Vamos ver vrias opes para recodificao, mas comeamos com o mais flexvel. Imagine-se que queremos corrigir alguns dos erros na tabela abaixo:

ex_recodificar <- tibble(ID=1:3,
                        Mes=c("janeiro","february","outubro"),
                        Ano=c(2013, 2009, 2015))

Queremos renomear february fevereiro para corrigir o erro de um gringo ignorante. Como? Temos duas tarefas: (i) identificar o caso relevante (e apenas o caso relevante), e (ii) especificar o novo valor. Em geral, quando queremos criar ou modificar uma varivel trabalhamos dentro de um mutate(), mas o mutate costuma mudar todos os valores na varivel. Aqui, queremos sobreescrever a varivel Mes com nova informao apenas no caso de february. Para isso, combinamos mutate() com case_when(), que controla os detalhes de modificao no mutate.

case_when() funciona com um sintxe especfica: Estabelecemos vrias condies (do mesmo tipo que usamos no filter()), e depois de cada uma usamos ~ e em seguida o novo valor para substituir quando esta condio se aplica. Por exemplo, em nosso caso queremos a condio Mes=="february" e o novo valor "fevereiro". Ento precisamos de Mes=="february"~"fevereiro".

ex_recodificar %>% mutate(Mes=case_when(Mes=="february"~"fevereiro"))

Qual o resultado? Mudamos o valor para fevereiro, est correto, mas apagamos os valores dos outros dois meses. O mutate() realmente muda a coluna inteira, ento as condies no case_when() precisam ser completas para todos os valores na nova coluna. Felizmente, existe um atalho para preservar os valores originais caso a condio anterior no seja satisfeita: Usamos uma condio que sempre satisfeita: a condio literalmente TRUE, e um novo valor que no muda nada; o nome da varivel original. Ento:

ex_recodificar %>% mutate(Mes=case_when(Mes=="february"~"fevereiro",
                                        TRUE~Mes))

Agora est correto. A sintxe um pouco especfica ento no se preocupe se receber uma mensagem de erro, volte para verificar se tem um ~ depois de cada condio, e uma vrgula entre cada recodificao. Tambm lembre-se que as condies precisam de dois sinais de igual.

Vamos recodificar a varivel Ano usando a mesma lgica, mas agora queremos gerar uma varivel binria que identifica todas as observaes depois de 2010. Tambm no queremos sobreescrever a varivel original, queremos uma nova varivel:

ex_recodificar %>% mutate(Depois_2010=case_when(Ano>2010~1,
                                                Ano<=2010~0))

importante que as duas condies aqui cubram todos os valores de Ano possvel.

Note que o case_when() flexvel demais: Podemos usar muitas condies, fazer referncia a todas as variveis na condio, e pegar os dados de todas as variveis para definir o novo valor. Para ilustrar, veja o exemplo bobagem abaixo:

ex_recodificar %>% mutate(Nova_Variavel=case_when(Ano>=2014~"Sim",
                                                  Ano<2014 & Mes=="janeiro"~Mes,
                                                  Ano<2014 & Mes=="february"~"No"))

Tome cuidado na construo das condies: Idealmente elas devem ser completas e tambm mutualmente excludentes. Se no, a ordem das condies importa e pode gerar resultados inesperados.

Ordenando os nossos dados

At agora, ignoramos a ordem das observaes em nosso banco de dados. A ordem importa por dois motivos. Primeiro, para apresentao de tabelas mais organizadas no documento final. Segundo, para facilitar funes especficas quando as nossas observaes tm uma ordem natural.

Ordenar: Arrange

O verbo para ordenar os dados Arrange(), e podemos especificar uma sequncia de variveis para ser usadas na ordenao. Vamos comear com uns dados desordenados:

ex_arrange <- tibble(ID=1:10,
                     Ano=c(2008, 2005, 2009, 2006, 2006, 2007, 2008, 2005, 2008, 2005),
                     Mes=c("Abril","Novembro","Julho","Maro","Novembro","Fevereiro",
                           "Junho","Novembro","Janeiro","Outubro"),
                     Valor=c(750,800,300,500,850,450,600,450,700,350))

Se quisermos as observaes em ordem de ano e valor:

ex_arrange %>% arrange(Ano, Valor)

O resultado que os dados so ordenados por Ano primeiramente, e dentro de cada Ano, por Valor.

Se quisermos as observaes em ordem decrescente, usamos um menos (-) para inverter a ordem, por exemplo, em ordem decrescente de valor:

ex_arrange %>% arrange(-Valor)

Filtrar para os Maiores/Menores: top_n e top_frac

Um tipo de filter() especfico quando queremos as maiores/menores observaes de acordo com uma varivel. Se no for um limite absoluto (ex. acima de 140), mas acima de um limite relativo (ex. os cinco maiores), precisamos fazer um ranquemanto e depois filtrar baseado nesse ranqueamento. possvel calcular isso manualmente, mas a maneira mais fcil usar a funo top_n(). Ela exige dois argumentos: o nmero de observaes desejadas, e a varivel na qual queremos aplicar o filtro.

ex_arrange %>% top_n(5, Valor)

Para os menores valores, usamos um menos (-) na frente do nmero de observaes.

ex_arrange %>% top_n(-5, Valor)

Existe uma funo bem parecida que devolve a proporo de observaes desejada: top_frac(). Para os maiores 30% de observaes, usamos o argumento de 0.3.

ex_arrange %>% top_frac(0.3, Valor)

Dados de Time-Series: lag e lead

Com os nossos dados ordenados, podemos implementar vrias tcnicas de anlise de Time Series (sries temporais). Por exemplo, podemos calcular, para cada observao, o valor de uma varivel no perodo anterior usando lag(). Note que essencial que os nossos dados j esto ordenados com arrange() antes de executar o lag. Dado que estamos construindo uma nova varivel, tambm necessrio usar lag dentro de mutate:

ex_arrange %>% arrange(Ano) %>% 
  mutate(Valor_anterior=lag(Valor))

Uma dificuldade com esta operao que no existe um valor anterior para a primeira observao - veja que o R insere um NA para o primeiro valor de Valor_anterior.

Com lag, fcil calcular a diferena entre Valor e Valor_anterior para focar em mudanas ao longo do tempo na varivel:

ex_arrange %>% arrange(Ano) %>% 
  mutate(Valor_anterior=lag(Valor),
         Valor_diferenca=Valor-Valor_anterior)

A operao de lead() equivalente, mas traz o valor subsequente.

ex_arrange %>% arrange(Ano) %>% 
  mutate(Valor_posterior=lead(Valor))

Use de novo o banco de dados flights:

  1. Ordene o banco de dados da menor maior durao (air_time), incluindo apenas os voos com destino de Anchorage (ANC).

flights %>% filter(dest == "ANC") %>% arrange(year, month, day, dep_time) %>%

Factors

Alm de nmeros, caracteres e lgicos (TRUE/FALSE), o tipo de dado mais til para anlise de dados um factor. Factors so variveis que podem assumir um nmero de valores limitados e pr-definidos (normalmente strings de texto). Pense em variveis categricas, nominais, ordinais etc. Factors nos ajudam a evitar erros e refletir a estrutura correta das nossas variveis, mas elas exigem uma disciplina e um cuidado por nosso lado.

Por exemplo, lembre-se do problema quando o nosso banco de dados incluiu february em vez de fevereiro? Este problema impossvel com uma varivel do tipo factor, pois definimos do incio os valores aceitveis num formato padro - os levels do factor. Vamos recriar o tibble ex_arrange de acima, mas agora especificamos o ms como uma varivel factor, com os levels permitidos pela natureza da varivel:

ex_arrange <- tibble(ID = 1:10, Ano = c(2008, 2005, 2009, 2006, 2006, 2007, 
    2008, 2005, 2008, 2005), Mes = factor(c("Abril", "Novembro", "Julho", 
    "Maro", "Novembro", "February", "Junho", "Novembro", "Janeiro", "Outubro"), 
    levels = c("Janeiro", "Fevereiro", "Maro", "Abril", "Maio", "Junho", 
        "Julho", "Agosto", "Setembro", "Outubro", "Novembro", "Dezembro")), 
    Valor = c(750, 800, 300, 500, 850, 450, 600, 450, 700, 350))

Bem mais trabalhoso, no?? Mas vale a pena: veja a tabela que isso produziu - em lugar de February o R colocou NA porque February no est na lista de levels que delimitamos no argumento levels. Isso impede que a nossa varivel tome valores impossveis, deixando a nossa anlise subsequente mais confivel. Lembre-se de incluir todos os valores possveis nos levels para evitar problemas futuros. Por exemplo temos Maio na lista de levels mesmo que ele no exista nos dados atuais.

O cdigo acima mostra como criar variveis de tipo factor num novo banco de dados. comum j termos um banco de dados com uma varivel de tipo caractere, e querermos transform-la em factor. Vamos criar o mesmo banco de dados de novo, mas sem especificar o ms como factor, e depois aplicar a transformao dentro de mutate.

ex_arrange <- tibble(ID = 1:10, Ano = c(2008, 2005, 2009, 2006, 2006, 2007, 
    2008, 2005, 2008, 2005), Mes = c("Abril", "Novembro", "Julho", "Maro", 
    "Novembro", "February", "Junho", "Novembro", "Janeiro", "Outubro"), 
    Valor = c(750, 800, 300, 500, 850, 450, 600, 450, 700, 350))

ex_arrange <- ex_arrange %>% mutate(Mes=factor(Mes))

Note que aqui no temos que especificar os levels do factor - o R usa os valores atuais para preencher os levels. Podemos fazer assim, mas lembre-se que os levels no incluem todos os meses, o que pode gerar problemas no futuro se quisermos adicionar mais dados.

Para permitir todos os doze meses desde o incio, temos que especificar os levels explicitamente:

ex_arrange <- tibble(ID = 1:10, Ano = c(2008, 2005, 2009, 2006, 2006, 2007, 
    2008, 2005, 2008, 2005), Mes = c("Abril", "Novembro", "Julho", "Maro", 
    "Novembro", "February", "Junho", "Novembro", "Janeiro", "Outubro"), 
    Valor = c(750, 800, 300, 500, 850, 450, 600, 450, 700, 350))

ex_arrange <- ex_arrange %>% mutate(Mes = factor(Mes, levels = c("Janeiro", 
    "Fevereiro", "Maro", "Abril", "Maio", "Junho", "Julho", "Agosto", 
    "Setembro", "Outubro", "Novembro", "Dezembro")))

Factors Ordenados

Existem mais duas vantagens em trabalhar com Factors. Primeiro, que podemos especificar uma ordem para a varivel mesmo para variveis que contm caracteres. Meses, por exemplo, so ordenados, mas no conseguimos usar na seo anterior para ordenao porque o R no entende automaticamente a ordem de Janeiro,Fevereiro. Na verdade, o R entendo que a ordem de caracteres alfabtica, que no faz sentido para a varivel de ms. Dentro de factor, podemos especificar ordered=TRUE para definir a ordem dos valores na sequncia que eles aparecem no vetor levels():

ex_arrange <- tibble(ID = 1:10, Ano = c(2008, 2005, 2009, 2006, 2006, 2007, 
    2008, 2005, 2008, 2005), Mes = factor(c("Abril", "Novembro", "Julho", 
    "Maro", "Novembro", "Fevereiro", "Junho", "Novembro", "Janeiro", "Outubro"), 
    levels = c("Janeiro", "Fevereiro", "Maro", "Abril", "Maio", "Junho", 
        "Julho", "Agosto", "Setembro", "Outubro", "Novembro", "Dezembro"), 
    ordered = T), Valor = c(750, 800, 300, 500, 850, 450, 600, 450, 700, 
    350))

Agora, podemos ordenar os dados por Ano e Ms:

ex_arrange %>% arrange(Ano, Mes)

Completando Bancos de Dados Parciais: complete

A segunda vantagem de trabalhar com factors que um factor deixa fcil preencher os valores ausentes da varivel. Por exemplo, os nossos clculos de lag, lead e diferena acima no fazem muito sentido - comparamos por exemplo o valor de Junho com o valor de Abril, pulando Maio porque ele no aparece no banco de dados originais. Ento o nosso clculo de lag inconsistente; um lag de um ms em alguns casos, dois meses em outros, 9 meses em outros. Um clculo consistente e confivel tem que refletir os dados ausentes explicitamente.

Podemos criar uma sequncia completa facilmente se as variveis originais foram criados corretamente como factors: Usamos o verbo complete() para preencher as variveis:

ex_arrange %>% complete(Ano, Mes)

Veja o resultado: Temos os 12 meses para cada um dos cinco anos, um banco de dados completo (e uma linha duplicada para Novembro 2005). A maioria dos valores so desconhecidos, NA porque no aparecem no banco de dados original, mas tudo bem, pelo menos entendemos onde temos dados e onde no, e no criamos lags e leads enganosos.

Note para o futuro que possvel especificar o valor que o complete usa para preencher observaes ausentes. Por exemplo, pode ser que o valor nos dados reflita vendas e que saibamos que no houve vendas nos meses ausentes do banco de dados. Neste caso, faz sentido preencher as observaes dos outros meses com zero usando o argumento fill:

ex_arrange %>% complete(Ano, Mes, fill=list(Valor=0))

Esta capacidade tambm ajudar muito na construo de grficos em tutoriais futuros: Valores podem desaparecer da legenda, e cores podem ficar inconsistentes entre grficos se no mantivermos estveis os levels do factor, ento importante aprender como a trabalhar com eles do incio.

Habilidade Bsica de Programao: Dados Ausentes, NA

Muitos problemas e erros so gerados pelo tratamento inapropriado de dados ausentes. Vamos revisar isto vrias vezes, mas por enquanto temos que reconhecer que NA um pedao de informao importante que no podemos ignorar. Jogar fora os dados ausentes significa perder muitas observaes potenciais, o que pode gerar um vis de seleo em nossa anlise. Ento sempre importante deixar os NAs explcitos.

Por outro lado, um NA ambguo e pode significar duas coisas diferentes: (i) Um NA estrutural que impossvel medir. Por exemplo, o PIB do Estado do Tocantins em 1970 (o estado foi criado apenas em 1988). (ii) Um NA que podia ter sido medido, mas por algum motivo no foi. Por exemplo, faltamos dados de 2014 porque os arquivos foram perdidos em um incndio. O tipo de NA vai afetar a nossa interpretao e tratamento de dados.

Trabalhando com dados NA exige algumas regras especficas. Para testar se um valor NA, no podemos usar o padro x=="NA". Temos que usar is.na(x), uma funo dedicada para testar se um valor NA ou no.

Se quisemos criar valores NA, temos que usar uma verso apropriada para o tipo de dado da varivel: existe NA_character_, NA_real_ e NA_integer_. Para subtituir NA com outro valor, use a funo replace_na(). Todos so demonstrados no exemplo abaixo.

ex_arrange %>% complete(Ano, Mes, fill=list(Valor=0)) %>%
  mutate(ID_NA=case_when(is.na(ID)~1,
                         TRUE~0),
         Valor=case_when(Valor==0~NA_real_,
                          TRUE~Valor),
         ID=replace_na(ID, "Desonhecido"))

Recodificao de levels de Factors

Um desafio em trabalhar com Factors que eles exigem funes especficas para recodificao dos seus levels. Existe um pacote inteiro para isso, que se chama forcats e parte do tidyverse.

Existem trs funes super-teis para trabalhar com factors, todas que comeam com fct_, e usamos todas dentro de um mutate(), dado que estamos mexendo com uma coluna que j existe.

Primeiro, quando queremos renomear os levels de um factor, usamos fct_recode. Por exemplo:

ex_arrange %>% mutate(Mes=fct_recode(Mes,
                                     "abril"="Abril",
                                     "junho"="Junho"))

Segundo, quando queremos ajustar a ordem de um factor ordenado, usamos fct_relevel. Por exemplo, para inverter a ordem do ms:

ex_arrange %>% mutate(Mes = fct_relevel(Mes, c("Dezembro", "Novembro", 
    "Outubro", "Setembro", "Agosto", "Julho", "Junho", "Maio", "Abril", 
    "Maro", "Fevereiro", "Janeiro"))) %>% arrange(Ano, Mes)

Terceiro, quando queremos reestruturar/simplificar o factor para menos nveis, usamos fct_collapse. Por exemplo, podemos juntar os primeiros trs meses e descrever como o primeiro quarto do ano, Q1:

ex_arrange %>% mutate(Mes=fct_collapse(Mes,"Q1"=c("Janeiro","Fevereiro","Maro")))

Mais detalhes no cheatsheet de forcats aqui.

Usando os dados de flights de novo:

flights <- flights %>% mutate(origin = factor(origin))

flights %>% mutate(origin = case_when(origin == "EWR" ~ "Newark", TRUE ~

Identificando Casos/Valores nicos (distinct)

Quais valores existem em nosso banco de dados? Em uma varivel de 1.000 linhas, quais valores nicos existem? As nossas observaes so duplicadas?

Comeando com um tibble, podemos pedir os valores/casos nicos que existem no banco de dados usando o verbo distinct(). Nos parnteses, especificamos quais variveis queremos avaliar. Se especificarmos uma varivel, recebemos todos os valores que existem no banco de dados para esta varivel:

flights %>% distinct(origin)

Com mais de uma varivel, identificamos os conjuntos que existem no banco de dados. Por exemplo, imaginar que queremos saber quais companhias areas voam de quais aeroportos (com a ajuda de arrange para organizar o resultado):

flights %>% distinct(origin, carrier) %>%
  arrange(origin, carrier)

Finalmente, distinct nos ajuda a entender melhor a unidade de anlise no banco de dados, e a presena de observaes/informaes duplicadas que podem indicar erros ou contaminar anlises subsequentes. Por exemplo, em teoria deve ser impossvel ter mais de um voo no mesmo horrio, com o mesmo nmero da mesma companhia, da mesma origem ao mesmo destino. Podemos confirmar isso observando que o nmero de linhas na seguinte operao igual ao nmero de linhas no banco de dados inteiro:

flights %>% distinct(year, month, day, dep_time, carrier, flight, origin, dest)

Os avies so tambm identificados por um nmero de cauda (tailnum), e imaginando que o tailnum seja nico para cada avio, parece razovel assumir que o mesmo avio no est voando no mesmo horrio mais que uma vez:

flights %>% distinct(year, month, day, dep_time, tailnum)

Repare, o nmero de linhas aqui 334.067, menos do 336.776 no banco de dados inteiro. Isso significa que temos mais que um tailnum decolando no mesmo horrioEstranho. crucial saber isso para informar as nossas anlises. Com mais investigao, entenderemos que a diferena produzida pela presena de observaes de dep_time e tailnum que so NA, desconhecidos, e portanto aparecem como duplicados.
